## Summary

Fixed the issue where float inputs to `interval_days` args in predictor and outcome specs were being coerced into integers.

### Root Cause
The docstrings in the spec classes showed incorrect type hints: `Union[int, float, NoneType]` while the actual Pydantic Field definitions were `Optional[Union[int, float]]`. This mismatch confused Pydantic's schema generation, causing float values to be coerced to integers.

### Changes Made

**1. Fixed docstrings in `/testbed/src/timeseriesflattener/feature_spec_objects.py`:**
   - `TemporalSpec` class (line 359): Changed from `Union[int, float, NoneType]` to `Optional[Union[int, float]]`
   - `PredictorSpec` class (line 486): Changed from `Union[int, float, NoneType]` to `Optional[Union[int, float]]`
   - `TextPredictorSpec` class (line 564): Changed from `Union[int, float, NoneType]` to `Optional[Union[int, float]]`
   - `OutcomeSpec` class (line 656): Changed from `Union[int, float, NoneType]` to `Optional[Union[int, float]]`

**2. Updated tests in `/testbed/tests/test_timeseriesflattener/test_resolve_multiple.py`:**
   - `test_resolve_multiple_avg()`: Changed `interval_days=2` to `interval_days=2.5`
   - `test_resolve_multiple_latest()`: Changed `interval_days=2` to `interval_days=2.5`

This ensures that float values (like 2.5 days) are now correctly preserved in the type schema, allowing the generation of predictors/outcomes with non-integer lookbehind/ahead windows.
